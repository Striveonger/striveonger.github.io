Kubernetes 限制Pod资源

vuePress-theme-reco Mr.Lee    2015 - 2025

Kubernetes 限制Pod资源

Mr.Lee 2025-03-03 09:55:23 DockerKubernetes

Kubernetes 限制Pod 的可以使用资源的, 有三种限制方式

# ResourceQuota

作用于命名空间, 限制命名空间整体的使用资源

# 核心功能

  1. 限制计算资源总量 限制命名空间中所有 Pod 的 requests 和 limits 总和（如 CPU、内存）。
    • 例如：整个命名空间的 Pod 总 CPU 请求不超过 10核，总内存限制不超过 20GiB。
  2. 限制存储资源总量 限制命名空间中所有 PersistentVolumeClaim (PVC) 的存储总量。
    • 例如：所有 PVC 的总存储量不超过 100GiB。
  3. 限制对象数量 限制命名空间中某些 Kubernetes 对象的数量（如 Pod、Service、ConfigMap 等）。
    • 例如：最多允许创建 20个 Pod 或 5个 LoadBalancer 类型的 Service。
  4. 结合作用域（Scopes）细化限制 通过 scopes 字段，可以针对特定类型的资源或 Pod 进行配额限制。例如：
    • 仅限制 BestEffort（未设置资源请求的 Pod）或 NotBestEffort（设置了资源请求的 Pod）。
    • 限制终止中的 Pod（Terminating）或正在运行的 Pod（NotTerminating）。
apiVersion: v1
kind: ResourceQuota
metadata:
  name: pod-resource-quota
  namespace: fm
spec:
  hard:
	# 计算资源
    requests.cpu: "10"         # 所有 Pod 的 CPU 请求总和不超过 10核
    requests.memory: "20Gi"    # 所有 Pod 的内存请求总和不超过 20GiB
    limits.cpu: "20"           # 所有 Pod 的 CPU 限制总和不超过 20核
    limits.memory: "40Gi"      # 所有 Pod 的内存限制总和不超过 40GiB

    # 存储资源
    requests.storage: "100Gi"  # 所有 PVC 的存储请求总和不超过 100GiB

    # 对象数量
    pods: "50"                 # 最多允许 50个 Pod
    services.loadbalancers: "5" # 最多允许 5个 LoadBalancer 类型的 Service
    configmaps: "20"           # 最多允许 20个 ConfigMap

  # 作用域（可选）
  scopes:
    - NotBestEffort  # 仅限制设置了资源请求的 Pod（排除 BestEffort Pod）

# 作用域（Scopes）

ResourceQuota 可以通过 scopes 细化限制目标，常用作用域包括：

作用域 说明
BestEffort 仅限制未设置资源请求的 Pod（resources.requests 为空）。
NotBestEffort 仅限制设置了资源请求的 Pod。
Terminating 仅限制处于终止状态的 Pod（activeDeadlineSeconds >= 0）。
NotTerminating 仅限制未设置终止时间的 Pod。
PriorityClass 限制特定优先级（PriorityClass）的 Pod（需指定具体 PriorityClass 名称）。

# LimitRange

主要用于限制命名空间（Namespace）内 Pod 或容器（Container）的资源使用量

# 使用场景

  1. 防止资源滥用 避免因配置错误导致单个 Pod 消耗过多资源，影响其他服务。
  2. 标准化资源配置 为开发人员提供默认资源模板，减少配置遗漏。

# 核心功能

  1. 设置默认资源请求和限制
    • 当 Pod 或容器未显式指定资源请求（requests）或限制（limits）时，LimitRange 可以自动为其填充默认值。
    • 例如：为所有容器的 CPU 和内存设置默认的 requests 和 limits。
  2. 限制资源使用范围
    • 定义资源（CPU、内存等）的最小值（min）、最大值（max）和默认值（default）。
    • 确保 Pod 或容器的资源请求和限制在合理范围内。
  3. 限制存储资源（Storage）
    • 可以限制 PersistentVolumeClaim（PVC）的存储大小范围。
  4. 限制资源比例
    • 定义 maxLimitRequestRatio，限制 limits 与 requests 的比值，防止资源突发使用超出预期。
apiVersion: v1
kind: LimitRange
metadata:
  name: example-limitrange
  namespace: default
spec:
  limits:
    - type: Container
      default:
        cpu: "500m"
        memory: "512Mi"
      defaultRequest:
        cpu: "100m"
        memory: "256Mi"
      min:
        cpu: "50m"
        memory: "128Mi"
      max:
        cpu: "2"
        memory: "4Gi"
      maxLimitRequestRatio:
        cpu: 5
        memory: 4
    - type: Pod
      min:
        cpu: "100m"
        memory: "256Mi"
      max:
        cpu: "4"
        memory: "8Gi"
    - type: PersistentVolumeClaim
      min:
        storage: "1Gi"
      max:
        storage: "10Gi"

# 总结

  • LimitRange 管理单个 Pod/容器的资源范围（如最小/最大值、默认值）。
  • ResourceQuota 管理整个命名空间的资源总量。

两者通常结合使用：

  1. LimitRange 确保单个 Pod 的资源合理
    • 例如：每个容器的内存限制不能超过 4GiB。
  2. ResourceQuota 确保命名空间的总量可控
    • 例如：整个命名空间的内存限制总和不超过 40GiB
特性 LimitRange ResourceQuota
作用对象 单个 Pod/容器 整个命名空间
核心能力 设置默认值、限制资源范围 限制资源总量、对象数量
典型场景 防止单个 Pod 资源过载，标准化配置 多租户资源隔离，成本控制
互补性 确保微观资源合理 确保宏观资源可控
