DataWarehousing(dw: 数据仓库)

# 数据的获取与分析

数据的两个作用:

  • 操作型记录的保存
  • 分析型决策的制定

操作型系统(数据生产方): 比如 ERP, CRM 等. 操作型系统, 一般不会维护历史数据, 只需要修改数据, 以反应最新的状态.

分析型系统(数据消费方): 比如 DW, BI 等. 通常分析型系统, 要求保存历史环境.

# DW/BI 的业务需求

  1. 可以方便的存取信息. 内容必须易于理解. 对于业务用户来说, 数据需要有直观性.

    • 直观性: 不能仅针对开发人员. 数据结构与标识要符合业务用户的思维过程和词汇.
    • 业务用户能以各种形式分割, 合并, 分析数据.

  2. 必须以一致的形式展现信息. 数据必须是可信的.

    • 组织不同来源的数据
    • 实现数据清洗
    • 确保保数据质量
    • 数据在真正适合用户需要时发布
    • 数据可在不同数据源共用

  3. 必须能够适应变化

    • 用户需求
    • 业务环境
    • 数据

    设计DW/BI系统时, 要考虑使其能够方便的处理无法避免的变化.

    如果需要修改 DW/BI系统中的描述性数据(元数据)时, 在能以适当的方式描述变化. 且变化对用户透明.

  4. 必须能够及时展现信息

    • 在发布未做数据清洗的数据时, 预期与期待.

  5. 保证信息财产的安全

  6. DW/BI 的成功, 取决于接入的业务群体

# ODS

操作性数据(Operational Data Store) 简称 ODS

与数据仓库在物理结构上不同。ODS存储的是当前的数据情况，给使用者提供当前的状态，提供即时性的、操作性的、集成的全体信息的需求

ODS作为数据库到数据仓库的一种过渡形式，能提供高性能的响应时间，ODS设计采用混合设计方式。ODS中的数据是"实时值"，而数据仓库的数据却是"历史值"，一般ODS中储存的数据不超过一个月，而数据仓库为10年或更多

  • ODS直接存放从业务抽取过来的数据，这些数据从结构和数据上与业务系统保持一致，降低了数据抽取的复杂性
  • 转移一部分业务系统的细节查询功能，因为ODS存放的数据与业务系统相同，原来有业务系统产生的报表，现在可以从ODS中产生
  • 完成数据仓库中不能完成的功能，ODS存放的是明细数据，数据仓库DW或数据集市DM都存放的是汇聚数据，ODS提供查询明细的功能
  • ODS数据只能增加不能修改，而且数据都是业务系统原样拷贝，所以可能存在数据冲突的可能，解决办法是为每一条数据增加一个时间版本来区分相同的数据
