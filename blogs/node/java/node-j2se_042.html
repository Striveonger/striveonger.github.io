学习Java多线程与高并发, ThreadLocal引出的 强软弱虚 . 之前在看共享锁时, 看到记录共享锁重入次数的实际上是ThreadLocal对象….今天我们就来扒一扒ThreadLocal

# 强引用

package xyz.striveonger.demo.concurrent.chapter_04.reference;

import java.io.IOException;

/**
 * @description: 强引用的示例
 * @author: Mr.Lee
 * @date: 2021-06-02 04:06
 */
public class StrongReferenceExample {

    static class M {
        @Override
        protected void finalize() throws Throwable {
            // 为了可以显示出对象被回收了...仅此而已
            System.out.println("m is finalize....");
            super.finalize();
        }
    }

    public static void main(String[] args) throws IOException {
        System.out.println("创建强引用对象...");
        M m = new M();
//        System.out.println("拆毁引用");
//        m = null;
        System.out.println("执行 gc");
        // 手动 GC
        System.gc();
        System.in.read();

        /**
         * 行为解读:
         *    m 变量, 堆内存中的 M对象
         *    m = null时, m变量指向 M对象的指针消失..
         *    这时,M对象是没有引用指向的(被视为垃圾对象)
         *    在手动GC时, 就会清掉所有垃圾对象
         *    在对象销毁时, 就会调用咱们重写的 finalize 方法了
         */
    }
}

package xyz.striveonger.demo.concurrent.chapter_04.reference;

import java.io.IOException;
import java.lang.ref.SoftReference;
import java.util.concurrent.ConcurrentSkipListMap;

/**
 * @description: 软引用的示例
 * @author: Mr.Lee
 * @date: 2021-06-02 04:46
 */
public class SoftReferenceExample {

    private static final int SIZE_1M = 1024 * 1024;

    public static void main(String[] args) throws IOException {

        // -Xms=20M -Xmx=20M....JVM启动时, 内存分配了20M
        // soft 软引用指向15M的连续内存空间
        SoftReference<byte[]> soft = new SoftReference<>(new byte[SIZE_1M * 15]);
        // 输出软引用, 所指向的数组空间的内存地址
        System.out.println(soft.get());

        // GC 不会清除掉, 软引用的对象
        System.gc();
        System.out.println(soft.get());
        // 分配一个10M的数组空间
        byte[] bytes = new byte[SIZE_1M * 10];

        // 因为, JVM内存只有20M, 在分配bytes
        // GC发现内存不够了. soft 所指向的10M会被GC清掉
        System.out.println(soft.get());
        // 执行结果:
        // [B@2c6a3f77
        // null

    }
}

//        byte[] bytes = new byte[SIZE_1M * 10]; // 有强引用在时, 是不会被GC掉的

        System.out.println(weak.get());
    }
}

# 虚引用

package xyz.striveonger.demo.concurrent.chapter_04.reference;

import java.io.IOException;
import java.lang.ref.PhantomReference;
import java.lang.ref.Reference;
import java.lang.ref.ReferenceQueue;
import java.nio.channels.SelectableChannel;
import java.util.concurrent.TimeUnit;

/**
 * @description: 虚引用的示例
 * @author: Mr.Lee
 * @date: 2021-06-03 05:21
 */
public class PhantomReferenceExample {

    private static final int SIZE_1M = 1024 * 1024;

    private static final ReferenceQueue<byte[]> queue = new ReferenceQueue<>();

    // -Xms20M -Xmx20M JVM启动内存20M
    public static void main(String[] args) throws IOException, InterruptedException {

        PhantomReference<byte[]> reference = new PhantomReference<>(new byte[SIZE_1M * 10], queue);

        // 虚引用是没办法拿到的....一般用来处理堆外内存
        System.out.println("[main]: reference -> " + reference);
        System.out.println("[main]: bytes -> " + reference.get());

        new Thread(() -> {
            while (true) {
                Reference<? extends byte[]> r = null;
                if ((r = queue.poll()) != null) {
                    // 一般在这处理业务逻辑, 销毁堆外内存
                    System.out.println("[t1]: reference -> " + r);
                    System.out.println("[t1]: bytes -> " + r.get());
                    break;
                }
            }
        }, "t1").start();

        TimeUnit.SECONDS.sleep(1);
        System.gc();

        System.in.read();
        System.out.println("end...");
    }
}

reference 对象指向的是堆外的内存空间, 正常情况下, 我们是无法访问的.

PhantomReference 是要配合 ReferenceQueue 来使用的...

因为虚引用指向的内存空间在堆外, 当堆内空间不够用时, JVM会执行GC

在GC时, 也会把堆个内存中, 虚引用指向的对象, 一同清掉. 在清掉的同时, 会把Reference对象放入指定的队列中, 以通知, 销毁内存后要执行的业务逻辑.

# ThreadLocal

ThreadLocal 是线程独有的变量, 不存在共享的问题…………..

# 使用场景

从丙丙那里薅过来的

之前我们上线后发现部分用户的日期居然不对了, 排查下来是SimpleDataFormat的锅.

当时我们使用SimpleDataFormat.parse()方法, 内部有一个Calendar对象.

在调用SimpleDataFormat.parse()方法时, 会先调用Calendar.clear(), 然后调用Calendar.add().

如果 线程A 调用了Calendar.add(), 然后 线程A 挂起, 线程B 调用Calendar.clear(), 这时候parse()方法解析的时间就不对了.

其实要解决这个问题很简单, 让每个线程都new一个自己的 SimpleDataFormat就好了, 但是1000个线程难道new 1000个SimpleDataFormat?

ThreadLocal.get()

// ThreadLocal.get
public T get() {
    Thread t = Thread.currentThread();
    ThreadLocalMap map = getMap(t);
    if (map != null) {
        // 从当前线程的ThreadLocalMap中拿到Entry
        ThreadLocalMap.Entry e = map.getEntry(this);
        if (e != null) {
            T result = (T)e.value;
            return result;
        }
    }
    return setInitialValue();
}
// ThreadLocal.getMap
ThreadLocalMap getMap(Thread t) {
    // 线程自己维护了一个ThreadLocalMap对象
    return t.threadLocals;
}

// ThreadLocalMap.getEntry
Entry getEntry(ThreadLocal<?> key) {
    // 通过hash, 计算索引
    int i = key.threadLocalHashCode & (table.length - 1);
    // 从数组中查找
    Entry e = table[i];
    if (e != null && e.get() == key) {
        return e;
    } else {
        // hash 冲突了
        return getEntryAfterMiss(key, i, e);
    }
}

// ThreadLocalMap.getEntryAfterMiss
Entry getEntryAfterMiss(ThreadLocal<?> key, int i, Entry e) {
    Entry[] tab = table;
    int len = tab.length;
	
    // Entry 为空时, Map中没有呗, 直接返回null.
    while (e != null) {
        ThreadLocal<?> k = e.get();
        // 如果找到就返回呗~
        if (k == key) {
            return e;
        }
        if (k == null) {
            // 去掉失效的 entry
            expungeStaleEntry(i);
        } else {
            // 继续找下一个
            i = nextIndex(i, len);
        }
        e = tab[i];
    }
    return null;
}

// 游标向后挪一位呗
int nextIndex(int i, int len) {
    return ((i + 1 < len) ? i + 1 : 0);
}

// 初始化当前线程的 map 与 Value
// ThreadLocal.setInitialValue
private T setInitialValue() {
    // 模板模式, 调用 () -> new Context(0) 
    T value = initialValue();
    Thread t = Thread.currentThread();
    ThreadLocalMap map = getMap(t);
    if (map != null) {
        map.set(this, value);
    } else {
        // 创建属于当前线程的 ThreadLocalMap
        createMap(t, value);
    }
    if (this instanceof TerminatingThreadLocal) {
        TerminatingThreadLocal.register((TerminatingThreadLocal<?>) this);
    }
    return value;
}

public void set(T value) {
    Thread t = Thread.currentThread();
    ThreadLocalMap map = getMap(t);
    if (map != null) {
        map.set(this, value);
    } else {
        createMap(t, value);
    }
}

// ThreadLocalMap.set
private void set(ThreadLocal<?> key, Object value) {
    Entry[] tab = table;
    int len = tab.length;
    // 根据hashCode获取索引值
    int i = key.threadLocalHashCode & (len-1);
    
    // 如果 tab[i] == null, 那么说明这个位置还没有用过, 可以直接使用
    // hash冲突, 向后查找插入点
    for (Entry e = tab[i]; e != null; e = tab[i = nextIndex(i, len)]) {
        ThreadLocal<?> k = e.get();
        if (k == key) {
            // 修改已经有值
            e.value = value;
            return;
        }
        if (k == null) {
            // k 被GC回收了, k的位置, 就可以让出来了.
            // i位置, 塞入新的 k, v
            replaceStaleEntry(key, value, i);
            return;
        }
    }
    // 创建新的Entry对象
    tab[i] = new Entry(key, value);
    int sz = ++size;
    if (!cleanSomeSlots(i, sz) && sz >= threshold) {
        // 去除被GC回收的位置 & 扩容(简单粗暴 *2)
        rehash();
    }
}

// ThreadLocal.remove
public void remove() {
    ThreadLocalMap m = getMap(Thread.currentThread());
    if (m != null) {
        m.remove(this);
    }
}
// ThreadLocalMap.remove
private void remove(ThreadLocal<?> key) {
    Entry[] tab = table;
    int len = tab.length;
    // 通过hashCode计算索引位置
    int i = key.threadLocalHashCode & (len-1);
    // 处理hash冲突
    for (Entry e = tab[i]; e != null; e = tab[i = nextIndex(i, len)]) {
        // 判断是不是要删除的Entry对象
        if (e.get() == key) {
            e.clear();
            // 清理掉i位置上的Entry对象
            expungeStaleEntry(i);
            return;
        }
    }
}

ThreadLocal在保存的时候会把自己当做 Key 存在 ThreadLocalMap 中, 正常情况应该是key和value都应该被外界强引用才对, 但是现在key被设计成WeakReference弱引用了.

ThreadLocalMap 的每个 Entry 都是一个对 key 的弱引用, 这一点从super(k)可看出. 另外, 每个 Entry 都包含了一个对 value 的强引用.

ThreadLocal在没有外部强引用时, 发生GC时会被回收. 如果创建ThreadLocal的线程一直持续运行, 那么这个Entry对象中的value就有可能一直得不到回收, 发生内存泄露.

# 防止内存泄露-源码做出努力

其实在上面咱们不止一次看到了…

private void replaceStaleEntry(ThreadLocal<?> key, Object value, int staleSlot) {
    Entry[] tab = table;
    int len = tab.length;
    Entry e;

    int slotToExpunge = staleSlot;
    for (int i = prevIndex(staleSlot, len); (e = tab[i]) != null; i = prevIndex(i, len)) {
        if (e.get() == null) {
            slotToExpunge = i;
        }
    }

    for (int i = nextIndex(staleSlot, len); (e = tab[i]) != null; i = nextIndex(i, len)) {
        ThreadLocal<?> k = e.get();
        if (k == key) {
            e.value = value;

            tab[i] = tab[staleSlot];
            tab[staleSlot] = e;

            if (slotToExpunge == staleSlot)
                slotToExpunge = i;
            cleanSomeSlots(expungeStaleEntry(slotToExpunge), len);
            return;
        }

        if (k == null && slotToExpunge == staleSlot) {
            slotToExpunge = i;
        }
    }

    tab[staleSlot].value = null;
    tab[staleSlot] = new Entry(key, value);

    if (slotToExpunge != staleSlot) {
        cleanSomeSlots(expungeStaleEntry(slotToExpunge), len);
    }
}



private void expungeStaleEntries() {
    Entry[] tab = table;
    int len = tab.length;
    for (int j = 0; j < len; j++) {
        Entry e = tab[j];
        if (e != null && e.get() == null) {
            expungeStaleEntry(j);
        }
    }
}

private int expungeStaleEntry(int staleSlot) {
    Entry[] tab = table;
    int len = tab.length;

    tab[staleSlot].value = null;
    tab[staleSlot] = null;
    size--;

    Entry e;
    int i;
    for (i = nextIndex(staleSlot, len); (e = tab[i]) != null; i = nextIndex(i, len)) {
        ThreadLocal<?> k = e.get();
        if (k == null) {
            e.value = null;
            tab[i] = null;
            size--;
        } else {
            int h = k.threadLocalHashCode & (len - 1);
            if (h != i) {
                tab[i] = null;

                while (tab[h] != null) {
                    h = nextIndex(h, len);
                }
                tab[h] = e;
            }
        }
    }
    return i;
}

另外, 会在 rehash 方法中通过 expungeStaleEntry 方法将键和值为 null 的 Entry 设置为 null 从而使得该 Entry 可被回收.

通过这种方式, ThreadLocal 可防止内存泄漏.

# 防止内存泄露-编码时需注意

虽然源码层级, 做了处理. 但还是有内存泄露的风险…

用过之后, 就再也不用了…在不手动移除的情况下, 还是会发生内存泄露

























 



package xyz.striveonger.demo.concurrent.chapter_04.treadlocal;

/**
 * @description: 上下文对象
 * @author: Mr.Lee
 * @date: 2021-06-08 15:16
 */
public class Context<T> implements AutoCloseable {
    ThreadLocal<T> holds = new ThreadLocal<>();

    public Context(T context) {
        holds.set(context);
    }

    public void setContext(T context) {
        holds.set(context);
    }

    public T getContext() {
        return holds.get();
    }

    @Override
    public void close() throws Exception {
        holds.remove();
    }
}
