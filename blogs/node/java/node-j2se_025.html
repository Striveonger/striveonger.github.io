# 1. wait/notify

public class Basket {

    private int capacity = 0;
    private volatile int count = 0;
    private List<Object> containers = new ArrayList<>();

    final Object lock = new Object();

    public Basket(int capacity) {
        assert capacity > 0;
        this.capacity = capacity;
    }

    public int getCount() {
        return count;
    }

    public void put(Object o) {
        synchronized (lock) {
            try {
                while (count == capacity) {
                    lock.wait();
                }
                count++;
                containers.add(o);
                lock.notifyAll(); // 叫醒所有线程, 包括生产者线程
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }
    }

    public Object get() {
        synchronized (lock) {
            try {
                while (count == 0) {
                    lock.wait();
                    System.out.println(String.format("消费者 waiting... %d", count));
                }
                Object o = containers.remove(0);
                count--;
                lock.notifyAll(); // 叫醒所有线程, 包括消费者线程
                return o;
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }
        // 降级处理(哈哈~)
        return new Object();
    }
}

# 2. Condition

Condition对象中维护了自己的等待队列.

J.U.C包提供了Conditon接口，用以对原生的Object.wait()、Object.notify()进行增强。

Condition接口的实现类其实是在AQS中——ConditionObject，ReentranLock的newConditon方法其实是创建了一个AbstractQueuedSynchronizer.ConditionObject对象

public class Basket {
		private int capacity = 0;
    private volatile int count = 0;
    private List<Object> containers = new ArrayList<>();

    final ReentrantLock lock = new ReentrantLock();
    final Condition provider = lock.newCondition(); // 创建生产者等待队列
    final Condition consumer = lock.newCondition(); // 创建消费者等待队列

    public Basket(int capacity) {
        assert capacity > 0;
        this.capacity = capacity;
    }

    public int getCount() {
        return count;
    }

    public void put(Object o) {
        lock.lock();
        try {
            // 之所以不用if, 是因为provider解封后, 当前线程拿到执行资源时, 其他线程已经把count填满了, 所以依旧无法执行. 所以不能只判断一次
            while (count == capacity) {
                provider.await(); // 使当前线程, 进入生产者等待队列中等待
            }
            count++;
            containers.add(o);
            consumer.signalAll(); // 唤醒消费者等待队列里的所有线程
        } catch (InterruptedException e) {
            e.printStackTrace();
        } finally {
            lock.unlock();
        }
    }

    public Object get() {
        lock.lock();
        try {
            while (count == 0) {
                consumer.await(); // 使当前线程, 进入消费者等待队列中等待
            }
            Object o = containers.remove(0);
            count--;
            provider.signalAll(); // 通知所有生产者, 可以生产了
            return o;
        } catch (InterruptedException e) {
            e.printStackTrace();
        } finally {
            lock.unlock();
        }
        // 降级处理(哈哈~)
        return new Object();
    }
}

# 3. LockSupport

LockSupport类的核心方法其实就两个：park()和unpark()，其中park()方法用来阻塞当前调用线程，unpark()方法用于唤醒指定线程。 这其实和Object类的wait()和signal()方法有些类似，但是LockSupport的这两种方法从语意上讲比Object类的方法更清晰，而且可以针对指定线程进行阻塞和唤醒。

LockSupport类使用了一种名为Permit（许可）的概念来做到阻塞和唤醒线程的功能，可以把许可看成是一种(0,1)信号量（Semaphore），但与 Semaphore 不同的是，许可的累加上限是1。 初始时，permit为0，当调用unpark()方法时，线程的permit加1，当调用park()方法时，如果permit为0，则调用线程进入阻塞状态。

/**
 * @author Mr.Lee
 * @description: LockSupport
 * @date 2021-04-21 9:20 PM
 */
public class LockSupportExample {

    public static void main(String[] args) {
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 10; i++) {
                System.out.println(i);
                if (i == 5) {
                    LockSupport.park(); // 车速太快, 我要停车
                }
                SleepHelper.sleepSeconds(1);
            }
        }, "t1");

        t1.start();

        LockSupport.unpark(t1);
        SleepHelper.sleepSeconds(7);
        System.out.println("main after 7s");
//        LockSupport.unpark(t1); // 继续开车
    }
}

# 4. CyclicBarrier

CyclicBarrier是一个辅助同步器类，在JDK1.5时随着J.U.C一起引入。

这个类的功能和我们之前介绍的CountDownLatch有些类似。我们知道，CountDownLatch是一个倒数计数器，在计数器不为0时，所有调用await的线程都会等待，当计数器降为0，线程才会继续执行，且计数器一旦变为0，就不能再重置了。

与CountDownLatch不同的是 CountDownLatch 阻塞的是mian线程(或调用方线程), 但CyclicBarrier阻塞的是工作线程

CyclicBarrier可以认为是一个栅栏，栅栏的作用是什么？就是阻挡前行。 顾名思义，CyclicBarrier是一个可以循环使用的栅栏，它做的事情就是： 让线程到达栅栏时被阻塞(调用await方法)，直到到达栅栏的线程数满足指定数量要求时，栅栏才会打开放行。

这其实有点像军训报数，报数总人数满足教官认为的总数时，教官才会安排后面的训练。

/**
 * @author Mr.Lee
 * @description: 围栏的使用和一些使用场景的探究
 * @date 2021-04-20 6:06 AM
 */
public class CyclicBarrierExample {

    private static final Queue<String> queue = new ArrayDeque<>(3);

    public static void main(String[] args) {
        // 齐聚首, 当把酒言欢
        CyclicBarrier barrier = new CyclicBarrier(3, () -> {
            // 模拟处理队列消息
            System.out.println(Thread.currentThread().getName() + ", Start Consumption...");
            while (!queue.isEmpty()) {
                System.out.println("queue item : " + queue.poll());
            }
            System.out.println(Thread.currentThread().getName() + ", End Consumption...");
        });

        Thread t1 = new Thread(() -> {
            // 模拟从网络爬取内容
            System.out.println(Thread.currentThread().getName() + ", start...");
            queue.add("html");
            try {
                barrier.await();
            } catch (InterruptedException | BrokenBarrierException e) {
                e.printStackTrace();
            }
            System.out.println(Thread.currentThread().getName() + ", end...");
        }, "t1");

        Thread t2 = new Thread(() -> {
            // 模拟从数据库中读一条记录
            System.out.println(Thread.currentThread().getName() + ", start...");
            queue.add("db");
            try {
                // 表示出诚意的同时, 也不失自己的底线. 时间过了, 就各玩各的~
                barrier.await(5, TimeUnit.SECONDS);
            } catch (InterruptedException | BrokenBarrierException | TimeoutException e) {
                e.printStackTrace();
            }
            System.out.println(Thread.currentThread().getName() + ", end...");
        }, "t2");
        Thread t3 = new Thread(() -> {
            // 模拟从硬盘中读取一个文件
            System.out.println(Thread.currentThread().getName() + ", start...");
            queue.add("file");
            try {
//                SleepHelper.sleepSeconds(10); // 他日再聚的结果
                SleepHelper.sleepSeconds(4); // 把酒言欢


                barrier.await();
            } catch (InterruptedException | BrokenBarrierException e) {
                e.printStackTrace();
            }
            System.out.println(Thread.currentThread().getName() + ", end...");
        }, "t3");

        t1.start();
        t2.start();
        t3.start();
    }
}

# 5. Semaphore

Semaphore，又名信号量，这个类的作用有点类似于“许可证”。有时，我们因为一些原因需要控制同时访问共享资源的最大线程数量，比如出于系统性能的考虑需要限流，或者共享资源是稀缺资源，我们需要有一种办法能够协调各个线程，以保证合理的使用公共资源。

Semaphore维护了一个许可集，其实就是一定数量的“许可证”。 当有线程想要访问共享资源时，需要先获取(acquire)的许可；如果许可不够了，线程需要一直等待，直到许可可用。当线程使用完共享资源后，可以归还(release)许可，以供其它需要的线程使用。

/**
 * @author Mr.Lee
 * @description: 限制线程数量
 * @date 2021-04-20 9:33 PM
 */
public class SemaphoreExample {


    public static void main(String[] args) throws Exception {

        final Semaphore semaphore = new Semaphore(2);
        Runnable runnable = () -> {
            try {
                semaphore.acquire();
                System.out.println(Thread.currentThread().getName() + ", start...");
                SleepHelper.sleepMilliSeconds(1000);
                System.out.println(Thread.currentThread().getName() + ", end...");
            } catch (InterruptedException e) {
                System.out.println("阻塞被打断...");
            } finally {
                semaphore.release();
            }
        };

        for (int i = 0; i < 5; i++) {
            new Thread(runnable, "t" + (i + 1)).start();
        }
    }
}

# 6. RateLimiter

限流的目的是通过对并发访问/请求进行限速或者一个时间窗口内的的请求进行限速来保护系统，一旦达到限制速率则可以拒绝服务或进行流量整形。

常用的限流方式和场景有：限制总并发数（比如数据库连接池、线程池）、限制瞬时并发数（如nginx的limitconn模块，用来限制瞬时并发连接数，Java的Semaphore也可以实现）、限制时间窗口内的平均速率（如Guava的RateLimiter、nginx的limitreq模块，限制每秒的平均速率）；其他还有如限制远程接口调用速率、限制MQ的消费速率。另外还可以根据网络连接数、网络流量、CPU或内存负载等来限流。

两个限流相关的基本算法：漏桶算法和令牌桶算法

/**
 * @author Mr.Lee
 * @description: 令牌桶的流控
 * @date 2021-04-20 6:49 AM
 * http://ifeve.com/guava-ratelimiter/
 */
public class RateLimiterExample {

    public static void main(String[] args) {
        // 每秒只发出5个令牌
        RateLimiter limiter = RateLimiter.create(5.0);
        SleepHelper.sleepMilliSeconds(200); // 桶中至少有一个令牌了
        for (int i = 0; i < 10; i++) {
            new Thread(() -> {
                boolean flag = limiter.tryAcquire();
                while (!flag) {
                    flag = limiter.tryAcquire(1, TimeUnit.SECONDS);
                    System.out.println(Thread.currentThread().getName() + ", 没有得到执行令牌");
                }
                System.out.println(Thread.currentThread().getName() + ", 执行结束");
            }, "t" + i).start();
            SleepHelper.sleepMilliSeconds(100); // 速率为200时, 刚刚好与生成令牌的速率一样. 不会轮空哦~
        }
    }
}

# 7. Exchanger

Exchanger——交换器，是JDK1.5时引入的一个同步器，从字面上就可以看出，这个类的主要作用是交换数据。

Exchanger有点类似于CyclicBarrier，我们知道CyclicBarrier是一个栅栏，到达栅栏的线程需要等待其它一定数量的线程到达后，才能通过栅栏。

/**
 * @author Mr.Lee
 * @description:
 * @date 2021-04-21 7:22 AM
 */
public class ExchangerExample {

    public static void main(String[] args) {
        Exchanger<String> exchanger = new Exchanger<>();
        new Thread(() -> {
            String s = "T1";
            SleepHelper.sleepSeconds(1);
            try {
                s = exchanger.exchange(s); // 阻塞, 等待另一个线程执行交换操作
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            System.out.println(Thread.currentThread().getName() + ", s = " + s);
        }, "t1").start();

        new Thread(() -> {
            String s = "T2";
            SleepHelper.sleepSeconds(2);
            try {
                s = exchanger.exchange(s);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            System.out.println(Thread.currentThread().getName() + ", s = " + s);
        }, "t2").start();
    }
}
